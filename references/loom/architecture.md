### 1. VS Code Extension Package Name

The VS Code extension should live in `packages/vscode/`. This aligns with the pattern of `packages/cli/`, `packages/app/`, etc. The extension's identifier in the marketplace would be `reslava.loom` (following the `@reslava/loom` npm scope).

### 2. Architecture Diagram: Package Relationships

```
┌─────────────────────────────────────────────────────────────┐
│                        Delivery Layer                        │
├───────────────┬─────────────────────────────────────────────┤
│     CLI       │                VS Code Extension            │
│ (packages/cli)│               (packages/vscode)             │
└───────┬───────┴───────────────────────┬─────────────────────┘
        │                               │
        │          calls                │          calls
        ▼                               ▼
┌─────────────────────────────────────────────────────────────┐
│                   Application Layer (app)                   │
│   Orchestration use‑cases: completeStep, finalize, etc.     │
└───────────────┬───────────────────────────────┬─────────────┘
                │                               │
                │ uses                          │ uses
                ▼                               ▼
┌───────────────────────────────┐ ┌───────────────────────────┐
│        Domain Layer (core)    │ │ Infrastructure Layer (fs) │
│  Entities, Events, Reducers,  │ │  Load/Save, Path Utils,   │
│  Body Generators, Validation  │ │  Link Index, Repositories │
└───────────────────────────────┘ └───────────────────────────┘
```

**Rules:**
- `cli` and `vscode` may **only** import from `app` and `fs` (and `core` for types).
- `app` may **only** import from `core` and `fs`.
- `core` may **only** import from itself.
- `fs` may **only** import from `core` and standard libraries.

### 3. Document Types and CLI Commands Table

| Document Type | Create Command | Other Commands |
| :--- | :--- | :--- |
| **idea** | `loom weave idea "<Title>" [--thread <name>]` | `finalize`, `rename` |
| **design** | `loom weave design <thread-id>` | `refine-design`, `finalize`, `rename` |
| **plan** | `loom weave plan <thread-id>` | `start-plan`, `complete-step`, `rename` |

**Note:** All creation commands use the verb `weave`. I will update any outdated references in the documentation to reflect this.

### 4. Detailed Workflow: Idea → Design → Plan → Done

This is the canonical Loom workflow. Let's walk through it step by step, showing the CLI commands, the events fired, the core functions called, the state transitions, and the files generated.

| Step | User Action (CLI) | Event Fired | Core Reducer Called | State Transition | Files Generated / Modified |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1. Idea** | `loom weave idea "Add Dark Mode"` | (none – creation) | `createBaseFrontmatter` | `status: draft` | `threads/add-dark-mode/new-{timestamp}-idea.md` |
| **2. Finalize Idea** | `loom finalize new-...-idea` | (none) | ID generation | `status: active` | File renamed to `add-dark-mode-idea.md` |
| **3. Create Design** | `loom weave design add-dark-mode` | `CREATE_DESIGN` | `designReducer` | `status: draft` | `threads/add-dark-mode/add-dark-mode-design.md` |
| **4. Activate Design** | (automatic on first edit or manual) | `ACTIVATE_DESIGN` | `designReducer` | `draft → active` | Frontmatter updated |
| **5. Refine Design** | `loom refine-design add-dark-mode` | `REFINE_DESIGN` | `designReducer` | `active → active`, `version++`, `refined: true` | Design version incremented; child plans marked `staled` |
| **6. Finalize Design** | `loom finalize add-dark-mode-design` | `FINALISE_DESIGN` | `designReducer` | `active → done` | Frontmatter updated |
| **7. Create Plan** | `loom weave plan add-dark-mode` | `CREATE_PLAN` | `planReducer` | `status: draft` | `threads/add-dark-mode/plans/add-dark-mode-plan-001.md` |
| **8. Activate Plan** | `loom start-plan add-dark-mode-plan-001` | `ACTIVATE_PLAN` | `planReducer` | `draft → active` | Frontmatter updated |
| **9. Start Implementing** | (same command, auto‑activates) | `START_IMPLEMENTING_PLAN` | `planReducer` | `active → implementing` | Frontmatter updated |
| **10. Complete Step** | `loom complete-step add-dark-mode-plan-001 --step 1` | `COMPLETE_STEP` | `planReducer` | Step marked done; if all steps done → `done` | Steps table updated |
| **11. Finish Plan** | (automatic on last step) | `FINISH_PLAN` | `planReducer` | `implementing → done` | Frontmatter updated |
| **12. Thread Done** | (derived) | — | `getThreadStatus` | `IMPLEMENTING` → `DONE` | — |

### 5. Gaps and Edge Cases Identified

| Gap / Edge Case | Proposed Mitigation |
| :--- | :--- |
| **Design creation from idea** currently requires manual `weave design`. Could be streamlined. | Future command: `loom promote idea <thread-id>` to automatically create design from idea. |
| **No command to manually mark plan as done** if steps are not used. | Add `loom finish-plan <plan-id>` command. |
| **No way to reorder steps** in a plan. | Deferred to `plan-step-id` feature. |
| **Blocked steps** are displayed but no command to "unblock" them. | The blocker resolution is manual; future `loom unblock-step` could help. |
| **Stale plans** are marked but no command to "refresh" them. | Future `loom refresh-plan <plan-id>` could regenerate from updated design. |

This detailed workflow confirms our architecture supports the happy path cleanly. The gaps are all post‑MVP enhancements.
