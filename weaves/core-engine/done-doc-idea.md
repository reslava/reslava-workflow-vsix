# Plan Done Documents — Post-Implementation Record

## Problem

After a plan is implemented, the implementation details live only in chat docs — informal, unstructured, hard to navigate. There is no clean retrospective record of what was built, what diverged from the plan, and what decisions were locked in during implementation.

## Idea

Introduce a `done` doc type: `{plan-id}-done.md`, generated by a `loom.closePlan` command at the end of a plan implementation session.

The done doc contains:
- What was implemented (steps completed, files touched)
- Decisions made during implementation that weren't in the plan
- Anything that diverged from the original design
- Open items or follow-up ideas surfaced during work

It is a clean retrospective, not a raw chat log. It lives alongside the plan in the `plans/` folder (or a `done/` subfolder). The originating implementation chat is linked via `parent_id`.

## Why

- Gives future contributors a fast read on "what actually happened" without wading through a full chat doc.
- Closes the loop between the design intent and the implementation reality.
- Feeds into the context summary (`-ctx.md`) — done docs are the most accurate source for "Key Decisions Made".

## Open questions

- First-class `type: done` in the schema, or a convention (plain markdown, no special type)?
- Generated by AI at plan close, or written by the user, or both?
- Should `loom.closePlan` also mark all remaining steps done and transition plan status to `done`?
